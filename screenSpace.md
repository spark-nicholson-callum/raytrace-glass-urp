# スクリーンスペースレイトレーシング

レイトレで一番入りやすい段階です。

レイトレの一番難しいポイントは、当たったメッシュの色を計算することだと思います。
ちょっと意外かもしれないですが、GPU はその情報を全く管理していません。
単純に当たったメッシュの ID しか管理していないのです。

つまり、テクスチャの参照、ライティング、影、ラスターの描画処理などのすべてをレイトレ用のシェーダーで再現しないといけません。

それはさすがに大変ですので、まずはすでにある色情報を使うのが楽です。
今まで描画したものを参照してガラスを描画します。

もちろん屈折、特に反射だと、これですべての色を取得できるわけではありません。
反射でカメラの後ろや、屈折で物の後ろなど、画面外の描画も必要になりますが、とりあえず簡単なやり方で進めます。

## レイの仕組み
今回はすごく簡単なレイの仕組みでいきます。
レイは固定で2ステップにします。

1. ガラスの中を通って、ガラスの後ろ側に当たります。
2. 屈折してガラスから出て、世界の何かに当たります。

結果には3パターンがあります。

### ガラスから出るときの全反射
屈折ができないケースで、リアルなガラスの描画にはとても大事なケースですが、今回は無視します。
実装中の時は、全反射のケースは固定の色を出力して後回しにしましたね。

ちなみに完璧な球体なら、このケースは絶対に起こりませんよ。

### 外のレイが何も当たらない
一番簡単なケースです。何も当たっていないのでスカイボックスを描画するべきですね。
これは単純に、シェーダーにスカイボックステクスチャの参照を渡して、レイの方向で参照すれば終わりです。

### 外のレイがメッシュに当たった
ここですね。
まず、私たちにはレイがどこに当たったかのワールド座標の情報があります。
欲しい情報のために、そのワールド座標に相当するスクリーン座標への変換が必要です。
Unity がクリップスペースまでの変換を用意してくれますが、そこからはちょっと自分で変換する必要があります。

```hlsl
float4 clipPos = mul(_ViewProjMatrix, float4(pos, 1.0));
float2 ndc = clipPos.xy / clipPos.w;
ndc.y = -ndc.y;
float2 hitUv = ndc * 0.5 + 0.5;
```
これを使ってスクリーンのテクスチャを参照して色情報を取得します。

ですが、これはレイが当たった場所ではないかもしれないです！
画面に描画したものの後ろにある可能性もあります。

カメラのデプステクスチャも参照して、ワールド座標に戻し、同じ場所かどうか確認できます。

## 画面外のオブジェクトの描画
詳細が知らいたいであれば[ソース](Runtime/Shaders/FallbackLens.compute)を参考するべきです。
基本的に Unity の URP 仕組みをできる限り再現して、スクリーンスペースの色情報と同じように表示が目的です。

もともとレイトレの処理が必要の想定で作ったレンダーパスこれはすごく簡単のはずです。
全部の描画処理がラスターシェーダーでも、レイトレシェーダーでも同じ関数を使うように書かれているはずです。
Unity の HDRP や UE5 を使ったら大分楽のはずですよー

### コンパクションの最適化
この処理がかなり重たいですし、レイトレ処理のピクセルの一部にしか必要ないです。
ので再度コンパクションをすれば最適化になるかと思います。

レイトレが必要なピクセルのコンパクションと同じように `AppendBuffer` に情報を追加して、別の1次元コンピュートシェーダで処理を行います。
